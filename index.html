C++ basic

C++基础

基础概念

数据的排布方式-大端模式和小端模式

  • 根据数据在内存中排布顺序不同，数据的排布可以分为大端模式和小端模式，大小端和芯片和编译平台有关，c中一般是小端，java中一般是大端，而网络传输数据一般是大端模式。
  • 大端模式: 数据的高位存放在低地址，数据的低位存放在高地址
  • 小端模式：数据的高位存放在高地址，数据的地位存放在低地址
  • 数字型数据如int a = 1234，左边为数据的高位，右边为数据的低位
  • 字符型数据如 char * a = “abcd”,左边为数据的低位，右边为数据的高位

指针

指针的含义和大小

  • 64位系统中，1 word = 8 byte。32位系统中，1 word = 4 byte。其中 1 byte = 8 bit，64=8*8，32=4*8，即系统的位数其实就是系统中一个字所占的bit数

  • 指针就是一个地址，指针的大小和cpu、操作系统位数、编译器有关、因为系统是向下兼容的，所以取这些变量里最小的值作为指针的大小。指针的大小通常为4byte或者8byte。指针的大小和指针指向的类型无关

运行结果:

  • 之所以要给指针指定类型，是因为不同类型的变量大小不同，指定类型后在通过该指针取值时，才知道取该地址的值后的一个1个byte还是2个byte还是多少byte。注意到void * a是没有类型的，因此不能取其指向值的大小，因为系统也不知道它指向值的大小。void * 通常用于传入别的指针，操作系统通过变量的起始地址和该变量的大小来取值，这是底层的基本思想。

指针的加减

运行结果:

  • 可以看到不同类型的指针+1所增加的字节是不同的，指针的加减和其指向的类型相关，加减都是根据其大小作为基本的运算单位来计算的。如对int * 来说+1会使其增加4byte，对char * 来说+1会使其增加1byte，和上面相同，void * 不支持加减运算，因为其指向的大小未知。

指针函数

什么是函数指针

  • 函数指针就是指向函数入口的指针(真就万物皆可指针)，和别的变量一样，函数也可以通过指针访问。第一次接触函数指针是在执行shellcode的时候，需要将一段字符串转换成函数并执行。程序来自某大佬的博客，这是一段linux的shellcode，程序要在linux下编译执行
  • 其中int (*func) ()就是声明一个返回值为int、参数为空的函数func的指针
  • 同理void* (*func) (int a)就是声明一个返回值为指针，参数为int a的函数func的指针
  • 编译
  • 注意到我们这里是将一个局部变量作为函数执行，这个变量是字符数组，该值存在栈中。一些操作系统在编译的时候默认会加上安全编译选项，如堆栈不可执行，因此我们这里要把这个选项关掉，通过-z execstack,如果编译时没有加上该选项，那么我这里会报段错误

  • 既然说到字符数组，可以想到前面指针数组内容里提到的，字符数组和字符串的存储区域不同，字符串存储于常量存储区且不可修改，字符数组作为局部变量存在于main的栈上，如果我们将shellcode从字符数组换成字符串，那么不需要关闭堆栈不可执行保护程序也能运行。

FileZilla的导出文件处理脚本

背景

花了几天从Filezilla中读取了服务器中要下载的文件路径
导出之后才知道只需要其中一部分格式的文件，总不能再跑一遍，filezilla中
又没法批量处理导入文本，只好自己上了。ps:filezilla可以设置过滤器只保留需要的文件，但那是在下载之前才有效。

方案

filezilla导出文件是xml格式，无非是找个xml库解析或者粗暴的
当成一般文本处理，xml解析就不用说了，主要是第二种。看一下导出文件格式，
每个要下载的文件包含多个子属性。只要匹配到文件名并将属于同一文件的其他属性行丢弃就行。又是一个遍历中的删除问题。由于是xml格式，因此很多属性关键字是一样的，因此不能简单的将每一行放入列表中通过列表值索引(list.index)，这样会索引到前面的相同的值。所以想法是生成一个key为索引值，value为字符串的字典。
遍历时维护一个临时列表，每次遇到关键字就将该行及后几行放入列表中。这样遍历
下一行时检查该行是否在临时列表中，在就continue。能极大的提高处理效率。这个列表在每次匹配到关键字时重置和bu’zai’，避免该列表太大拖慢速度，因为要处理的是几百兆的文本。(python list检索复杂度是O(n))。

实现

总结和遇到的坑

1.遍历中的跨元素操作，需要根据情况选择字典还是列表，尤其是涉及到列表的
遍历时删除操作(会改变list索引)，比较稳妥的方法是将需要的数据拿出来放到单独的list中。
2.遍历时跨索引值进行操作和索引(i+1)需要考虑越界问题。
3.filezilla导入只认utf-8编码，要么在写入新xml时就指定，要么手动设置为utf-8
否则会显示导入成功但是不会看见任何导入数据.
4.vim中设置文本编码格式:set fileencoding=utf-8

路由器的WAN和LAN

某天发现一台服务器（172.16.x.x）无法登陆，通过ping检查网络联通时发现
没有得到服务器回复，回复的地址是另一个ip（172.16.x.x）。便用：

检查一下数据包怎么走，结果发现这个地址就在我的下一跳，这不就是路由器？（当前我连接的是wifi）（192.168.x.x）
但一看这地址，是内网地址，但不是我当前路由器所在的内网。于是抽风了想通过arp看一下这货的的mac地址
（首先需要ping来获取mac地址）

发现地址表里没有这个地址，这才想起arp协议是二层协议，针对局域网，在局域网内，数据转发是通过MAC表进行的。
因而arp -a只能显示当前局域网的arp表。而我要找的服务器地址相对我来说是个外网地址。
所以这玩意到底是什么，上nmap

nmap给出的猜测结果是WAP，说明大概率是个无线接入点

没辙了，百度一下。

原来路由器确实有两个地址，内网（相对自身）和外网地址，分别对应路由器的LAN口和WAN口，即路由器内接入的设备通过
LAN来组成一个局域网，路由器的内网地址（如这里的192.168.x.x）为内网提供服务，而WAN口接入运营商提供的猫来连接互联
网，或者通过别的设备接入一个新的局域网，此时WAN口就是为出内网而服务的，WAN口的地址即为另一个局域网内的地址
（如这里的172.16.x.x）。 即我对172网络的服务器的ping操作得到的是路由器出口地址的回复。

然后我检查了一下服务器的网线..发现原来是网线松了。此时为了验证刚才百度的结果，我再次对服务器:

服务器在172网络，按说tracert后出口地址应该是WAN口才对，但很奇怪，我的下一跳仍然是路由器内网地址192
接着我试着tracert 谷歌的DNS服务器8.8.8.8，发现下一跳也还是路由器内网地址。

那么如何验证这个猜想呢，只好远程一下服务器，因服务器与路由器的WAN口在同一个网段内，通过服务器pingWAN操作，然后查一下服务器的apr缓存表就知道了。
结果WAN口MAC地址和LAN口的MAC地址一个是5a一个是5b，即它们是同一个路由器的两个不同网卡

记一次服务器ssh登陆修复

问题描述

1.服务器远程登陆报错: connection refused

解决过程

1.首先考虑网络问题，发现可以ping通,排除网络问题
2.因为有人在无法登陆之前已经登陆了，尝试重启服务器,重启之后问题还在。
3.重启无效又没法登陆，只能跑去机房上屏幕调试了。

生平第一次进机房处理毛病，插上就是图形登陆界面，输入密码发现登陆之后会黑屏，然后回到登陆界面。
我日，难道是系统炸了。既然图形界面上不了，那就上终端。ctrl alt f1 进tty。
tty能进，那就简单了，尝试检查一下ssh服务是不是开着：

netstat -l | grep ssh

发现没有ssh服务，那就重启服务：

sudo sevice sshd restart

报错：Job for ssh.service failed because the control process exited with error code
从这个报错来看ssh的进程应该还在才对，但是top和端口都找不到相关服务。没辙了，直接百度这个错误吧。

然后找到了这个老哥的blog：

csdn

发现一条命令：

/usr/sbin/sshd -T

没查过这个命令，但是估计-T应该是test吧。测试配置文件
运行一下，报错了，配置文件错误，问题找到了，和上面这个老哥一样

但是我不懂ssh的配置语法，该怎么改呢
再次百度这个错误

cnblog

估计是ssh更新或者某个人把配置文件给改错了，

错误配置：
PermitRootLogin prohibit-password yes

正确配置：
PermitRootLogin yes

然后重启ssh服务

sudo service ssh restart

大功告成，可以登陆了。

思考

1.善用搜索引擎，直接搜索错误信息
2.这之后发现在图形界面仍然无法登陆，估计是根本没有安装图形桌面（但登陆界面是图形化的）
或者管理员做了某些限制吧，这个奇怪的现象把我解决问题的想法打乱了一点，因为ssh无法登陆
应该不会影响直接登陆才对。
